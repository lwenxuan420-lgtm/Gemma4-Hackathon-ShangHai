# 技术报告 · Gemma 产线维修智能数据看板

> 参赛组名：**Cheng Repair AI Lab** ｜ 正式项目名称：**Gemma 产线维修智能数据看板**
> 参赛赛道：AI Agent ｜ 模型：Google **Gemma 4（`gemma-4-26b-a4b-it`）** ｜ 形态：可离线运行的静态前端 + 可选真实后端
> 在线演示：`https://cheng137930948-l.github.io/gemma4-repair-dashboard/` ｜ 代码仓库：本仓库（仓库历史名 slug：`gemma4-repair-dashboard`）

## 1. 一句话概述

把"产线维修课"每天靠人工 Excel 汇总的维修数据，做成一套 **Gemma 4 驱动的智能体看板**：实时计算修复率 / WIP / 超期 / 物料闭环，并由 Gemma 4 依据 `SKILL.md` 规则自动产出**结论前置、可执行**的管理汇报与异常预警，把"看数据"升级为"给决策"。

## 2. 背景与痛点

电子厂 PCBA 维修现场的真实痛点：

| 痛点 | 现状 | 业务后果 |
|---|---|---|
| 数据分散 | MES / EIP / 本地 Excel / 人工记录各一套 | 班组长每天手工拼表 1–2 小时 |
| 修复率不透明 | 进/出板靠事后统计 | 待修积压（WIP）发现滞后 |
| 超期无预警 | 未结天数靠人看 | 老化板卡拖到 P0 才暴露 |
| 汇报靠人写 | 主管汇报口径不一 | 结论主观、改善建议空泛 |

目标：用一个 **Agent + Skill** 的方案，让"取数 → 算指标 → 出结论 → 提预警"端到端自动化。

## 3. 模型选型与系统架构

### 3.1 模型选型理由：为何选择 Gemma 4 的 `gemma-4-26b-a4b-it`（26B MoE）

参赛要求基于 Gemma 4（2B / 4B / 26B MoE / 31B Dense）开发。本项目是一个**需要多轮自主函数调用**的 Agent，对"指令遵循 + 工具协议格式稳定性 + 工具结果归纳"要求较高，因此在四种规格中选择 **`gemma-4-26b-a4b-it`（26B 总参数、约 4B 激活的 MoE 指令微调版）**，理由如下：

1. **能力下限要够**：Agent 要稳定输出 ```` ```tool_code ````、解析多轮 ```` ```tool_output ````、再按 `SKILL.md` 的 9 段式结构产出"结论前置"的汇报。实测 2B / 4B 稠密小模型在多轮函数调用中**容易丢失工具协议格式、对长工具结果归纳不稳**；26B 级别的推理与指令遵循能力，是稳定跑通函数调用循环的能力下限。
2. **MoE 的性价比最优**：`a4b` 表示**推理时只激活约 4B 参数**——以接近 4B 的时延 / 算力开销，换来接近 26B 的推理质量。本场景"单次报告要 3–5 轮工具调用、且现场演示对时延敏感"，MoE 正好命中"既要质量又要可接受时延"的甜点区。
3. **不选 31B Dense**：31B 稠密版能力更强，但**每个 token 都要激活全部 31B 参数**，时延与显存成本显著更高；对"结构化取数 + 汇报归纳"这类任务边际收益有限，不足以抵消时延代价，反而拖累现场零等待演示。
4. **选 `-it` 指令微调版**：相比 base 版，instruction-tuned 版本对"遵守 tool_code 协议 + 按 SKILL 规则输出结构化汇报"的**指令遵循显著更好**，是函数调用稳定性的关键。
5. **开放权重 → 可私有化 / 离线**：Gemma 4 是开放权重模型，未来可在**产线内网私有化部署（生产数据不出厂）**，契合工厂对数据隐私的硬约束；26B MoE 的规模也适合单卡 / 小集群本地部署，为后续端侧 / 边缘演进留出空间。

> 一句话：在"能稳定跑通多轮函数调用"的前提下取**最低时延成本**——这正是 26B-MoE-a4b 相对 2B / 4B（能力不足）与 31B-Dense（时延过高）的最优折中。

两条链路同源：前端 Demo 与后端真实链路共用同一套工具契约（`get_wip` 等返回 `{summary, data}`）与同一份 `SKILL.md`，保证"离线可演示、在线可复现"，且口径一致（修复率均为 81%）。

## 4. AI Agent 设计（核心）

本项目的智能体能力分两层，**同一套工具契约**贯穿前后端：

**（A）后端真实链路 —— Gemma 4 原生函数调用循环（`backend/agent.py` + `backend/tools.py`）**

由 **Gemma 4 自主决定调哪个工具、按什么顺序调**，后端不硬编码编排顺序：

1. **工具注册表（`tools.py`）**：把看板能力封装成 Python 函数 `get_wip`、`get_defect_detail`、`get_repair_trend`、`get_attendance`、`get_material_*` 等，统一返回 `{summary, data}`。
2. **函数调用循环（`agent.py` / `run_agent`）**：用 Gemma 文档的函数调用约定，把工具清单注入 system prompt；模型输出 ```` ```tool_code get_wip() ``` ````，后端用 `parse_tool_call` 解析、执行 Python 工具、再以 ```` ```tool_output ``` ```` 回灌；如此多轮 `user/model` 对话，直到模型信息足够、停止调用并输出最终汇报。
3. **记忆与轨迹（trace / run-log）**：每一步「选择了哪个工具、入参、得到的中间结论」记录在 `trace` 并落盘为 `output/agent_run_*.log`，即参赛要求的"代理工具调用逻辑 + 运行日志"。
4. **健壮性**：未知工具 / 解析失败会把错误回灌给模型让其自我纠正；达到 `max_turns` 上限强制收尾；整体失败时回退单轮纯文本，保证演示不中断。
5. **结构化产出**：按 `SKILL.md` 的 9 段式结构（摘要 / 口径 / 指标 / 预警 / 风险 / 原因 / 建议 / 跟进 / 数据缺口）输出。

**（B）前端离线 Demo 层（`gemma_adapter.js`）**：用相同的 tool-registry + trace 设计，在无网络、无 Key 时也能展示等价的多步推理过程，便于离线评审。

> 示例：综合看板把工单状态聚合成 WIP 板（总量 / 机种分布 / 老化分桶 / 超期清单 / 积压趋势 / 修复率）；真实链路下 Gemma 4 自行先调 `get_wip`、再调 `get_defect_detail`，最终给出结论："近期闭环工单修复率约 **81%**（已修 13 / 报废 3），关注报废根因"——与修复率看板整体修复率口径（42/52≈81%）一致。

## 5. 关键技术：在 Gemma 4 上实现原生函数调用

生产环境原系统采用 **Google ADK（Agent Development Kit）** 的 Agent + function calling 模式：

```python
from google.adk import Agent
from google.adk.tools import skill_toolset
root_agent = Agent(model="gemini-2.5-flash",
                   tools=[skill_toolset.SkillToolset(skills=[repair_skill])])  # 依赖 API 级 function calling
```

但 **Gemma 4 是开放权重模型，在 Gemini API 上仅支持 `generateContent`，不提供 `tools=` 参数与 `system_instruction=`**，无法直接把 ADK 的 `model` 换成 `gemma-4-26b-a4b-it`。

本项目的做法不是"放弃函数调用"，而是 **按 Gemma 4 文档的函数调用约定，自己在后端实现一个真正由模型驱动的函数调用循环**（`backend/agent.py`）：

```python
from google import genai
client = genai.Client(api_key=GOOGLE_API_KEY)

# 把 SKILL.md + 工具清单注入 system prompt，进入多轮 user/model 对话
contents = [{"role": "user", "parts": [{"text": system_prompt}]}]
for _ in range(MAX_TURNS):
    resp = client.models.generate_content(model="gemma-4-26b-a4b-it", contents=contents)
    contents.append({"role": "model", "parts": [{"text": resp.text}]})
    call = parse_tool_call(resp.text)          # 解析 ```tool_code get_wip()```
    if not call:                                # 模型不再调用工具 → 最终汇报
        break
    name, args = call
    result = TOOLS[name](args)                  # 在 Python 中真实执行工具
    contents.append({"role": "user",            # ```tool_output 回灌，进入下一轮
                     "parts": [{"text": f"```tool_output\n{json.dumps(result)}\n```"}]})
```

模型自主决定调用 `get_wip`、`get_defect_detail` 等工具的时机与顺序，后端真实执行并回灌结果，循环直至模型给出最终结论——这是**模型驱动的多步函数调用**，而非把数据一次性塞进 prompt 的提示工程，也不是后端硬编码的固定流程。整个过程的工具调用轨迹与运行日志（`output/agent_run_*.log`）即本项目智能体能力的直接证据。

## 6. Skill 机制（`repair-report/SKILL.md`）

以 **Markdown Skill** 定义分析规则，作为 Prompt 文本注入，模型据此生成结构化汇报，关键约束包括：

- **禁止编造数字**：用户未提供的数据不得编造，无法确认须标注"数据未提供"。
- **字段识别与别名映射**：自动归一化工单号 / 未结天数 / 进出板时间 / 维修员等字段的多种叫法。
- **分级预警规则**：未结 >7/15/30 天 → P2/P1/P0；同一不良 ≥3/5 片 → P2/P1；当天未闭环影响维修 → P1。
- **改善建议必须可执行**：要有责任方向、优先级、建议完成时间与验证方式，杜绝"加强管理"式空话。

## 7. 数据与安全（脱敏方案）

面向公开参赛仓库做了系统性脱敏：

- 全量改用程序生成的**脱敏演示数据**（虚构机种 KSK/ABC/XYZ/DEF/GHJ、虚构维修员、虚构日期），三看板开箱即用。
- 移除内部代号、真实姓名、内网绑定服务（`localhost:8787` 默认关闭，靠 `RDC_ENABLE_BINDING_SERVER` 开关门控）、第三方接口默认值。
- 真实 API Key 仅存本地 `backend/.env`（`.gitignore` 排除），仓库仅留 `.env.example` 占位。
- Webhook 推送在 Demo 模式下只生成预览、不真实发送。

本方案并非纸面设想：其 Agent 架构（工具注册表 + `SKILL.md` 规则 + 函数调用循环）已在**真实电子厂产线维修现场落地运行**，日常服务维修班组。脱敏后的落地规模：

| 维度 | 真实落地规模（已脱敏） |
|---|---|
| 接入数据源 | 28 个看板 / 数据源同步（工时达成率、领退料、出勤、产出趋势等） |
| 日处理数据量 | 维修明细 3 万+ 条 / 日 |
| 异常闭环 | 单日 500+ 条工单异常自动分级（P0–P3），推送 Teams / 企业微信并跟踪「已推送 / 已处理 / 已恢复」 |

## 10. 如何复现

**离线演示（零配置）**：`python -m http.server 8123` → 打开 `http://127.0.0.1:8123/index.html` → 进入三看板。

**真实 Gemma 4（一键，二选一）**：
```bash
# 方式 A · 本地一键脚本（自动建 venv、装依赖、同时起后端:8001 + 前端:8123）
./start.sh            # 首次会生成 backend/.env，填入 GOOGLE_API_KEY 后重跑

# 方式 B · Docker（一条命令起后端+前端）
GOOGLE_API_KEY=你的Key docker compose up
```
启动后打开 `http://127.0.0.1:8123/index.html`，在「AI 智能分析中心」点「开始分析」即走真实 Gemma 4 原生函数调用循环（后端 `http://127.0.0.1:8001`）。

**或手动起后端**：
```bash
cd backend && pip install -r requirements.txt
cp .env.example .env   # 填入 https://aistudio.google.com/apikey 申请的 GOOGLE_API_KEY
python report_server.py            # http://127.0.0.1:8001（端口由 .env 的 PORT 决定）
```

## 11. 局限与展望

- **当前局限**：函数调用基于 Gemma 文档的 prompt 约定（解析 ```` ```tool_code ````）而非 API 级 `tools=`，因此对模型输出格式有一定容错依赖；Demo 模式为等价的规则模拟，便于离线评审。
- **下一步**：① 在函数调用循环上叠加"自反思 / 重规划"，让模型对中间结论不满意时主动改调工具；② 接入真实 MES/EIP 数据源（保留脱敏开关）；③ 把推送从预览升级为可审计的真实闭环（带审批）。
